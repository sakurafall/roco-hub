## 三、评审计划（Phase 0 关卡）

### 3.1 评审方式

- **异步评审**：每人在本地 / GitLab / GitHub 上对 7 份文档各给一次书面意见
- **集中会议**：2 小时，对分歧点逐条讨论拍板
- **产出物**：每份文档的"评审 checklist"逐项勾选 + 修订 v1.1

### 3.2 评审 checklist（总）

参与者：产品负责人、前端负责人、后端负责人、设计师（若有）、合规 Owner

- [ ] **产品战略层面**

  - [ ] MVP-A 的 P0/P1/P2 拆分合理（01）
  - [ ] 验证门槛 7 指标 + 止损策略认同（01）
- [ ] **数据与产品层面**
  - [ ] Spirit 数据模型覆盖所有页面需求（02 + 03）
  - [ ] 首批 150-200 只精灵的挑选标准认同（02）
  - [ ] 可信标签 4 级足够（02）
- [ ] **设计与体验层面**
  - [ ] 8 个核心页面信息架构合理（03）

  - [ ] API 契约覆盖 MVP-A 全部需求（06）

## 六、文档维护规范

### 6.1 变更流程

- 任何变更走 PR + 至少 1 个评审
- 变更需更新对应文档的"本文件版本"与 `CHANGELOG.md`

### 6.2 版本号

- 每份文档末尾维护 `v1.0 / v1.1 / ...`
- 重大变更（结构/核心契约）：小版本升级（v1.0 → v2.0）
- 修订类变更：补丁升级（v1.0 → v1.1）

### 6.3 Phase 复盘

每个 Phase 结束：

- 在对应文档末尾追加一段"Phase X 复盘"
- 记录：实际 vs 预期 / 学到什么 / 下一步调整
